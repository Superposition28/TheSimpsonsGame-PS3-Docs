The Simpsons Game (2007) EA .mus Audio Format

File Extension: .mus

Origin: The Simpsons Game (PS3/Xbox 360), EA Redwood Shores.

Purpose: Container archives for interactive, segmented, and layered background music.

1. What We think

  • It is an Archive, Not a Stream: .mus files are not single continuous audio tracks. They are containers holding multiple distinct audio segments and layers. Treating them as a single raw stream results in corrupted audio (static/noise).
  • Big-Endian Table of Contents (TOC): The archive structure relies on a directory/TOC that begins at offset 0x08.
  • Fixed-Size TOC Entries: Each entry in the TOC is 12 bytes (0x0C) long and contains the following Big-Endian fields:
    • 0x00 (uint16): IDX (Stream index number)
    • 0x02 (uint16): DUMMY (Unknown/Padding)
    • 0x04 (uint32): HEADERPOS (Pointer to the entry's metadata word)
    • 0x08 (uint32): OFFSET (Absolute pointer to where the audio payload begins)
  • Implicit File Sizes: The size of an individual audio segment is calculated by subtracting its OFFSET from the OFFSET of the next entry in the TOC (or EOF for the final entry).
  • Block-Based ADPCM: The audio payloads are encoded in an EA ADPCM variant (EA-XAS for PS3, potentially XMA2 for Xbox 360). These codecs rely on strict frame/block alignments.
  • Padding Behavior: The sequence 0C 00 00 00 followed by nulls is used as padding to align blocks. Crucially: Stripping this padding destroys the internal ADPCM frame alignment, causing decoders like vgmstream to lose sync and output maximum-amplitude static.
  • Headerless Payloads: The raw data starting at OFFSET contains no self-describing headers. It cannot be decoded on its own.

2. Current Assumptions & Working Theories

  • The "Synthetic Header" Requirement: To properly decode an extracted payload, we assume a 32-byte (0x20) standard EA .xas header must be artificially constructed and prepended to the raw bytes.
  • The INFO Word: The 32-bit value located at an entry's HEADERPOS is assumed to be the critical metadata (channels, sample rate, or block layout flags) required to build the synthetic .xas header.
  • Companion Metadata Files (The "Missing Half"): We assume the .mus file only acts as a bulk reservoir of audio fragments. The logic for how these fragments are stitched together, layered, and looped during gameplay is stored in external MPF-variant metadata files (specifically .msb and .msx files found in the game directories).
  • Subtype Signatures: The first few bytes of the file (e.g., 0F 00 00 00 78 01 32 00 vs 0B 00 00 00 02 03 02 03) dictate format versioning or routing logic, though the exact routing differences between "Type 1" and "Type 2" remain unmapped.

3. What We Don't Know (Pending Investigation)

  • The Bit-Level Structure of the INFO Word: We know HEADERPOS points to metadata, but we do not yet know the exact bitmask of this 32-bit word (e.g., which bits represent sample rate vs. channel count).
  • Internal MSB/MSX Structure: While we know .msb/.msx files control the interactive playback map, we do not know their internal grammar, nor how they reference the specific IDX values inside the .mus archives.
  • Layer Multiplexing: It is unconfirmed whether multi-channel stems (e.g., Layers 1, 2, and 3) are interleaved within a single TOC payload, or if they are stored as separate consecutive TOC entries that the engine plays simultaneously.
  • The Unknown Integer at 0x04: The purpose of the integer at offset 0x04 of the main file header (previously assumed to be a "cue count" ranging from 6 to 249) is not definitively proven, though it likely correlates with the total number of valid TOC entries.
